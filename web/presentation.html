System architecture

Three hydraulic pistons simulated with an HMI (master) and piston client (slave) communicating over a Fake EtherCAT-style bus.

  • HMI / Server: Tkinter or Streamlit UI to send start/single/stop and timing values.
  • Piston Client: Consumes commands, runs extend/retract sequence, publishes state.
  • Fake EtherCAT Bus: Thread-safe mailbox for commands and states.
  • LogBuffer: Collects server/client messages; shown side-by-side on the web HMI.

Communication protocol

Commands flow master → slave, state flows slave → master; JSON-like dicts over the Fake EtherCAT bus.

{
  "type": "start",          // start | single | stop
  "latched": true,          // auto-repeat after completion
  "durations": [[1.5,1.0],[2.0,1.0],[2.5,1.0]]
}
// Client → HMI
{
  "status": "running|complete|stopped",
  "active_piston": 0,
  "direction": "extend|retract",
  "remaining_ms": 850,
  "cycle_count": 3,
  "latching": true,
  "message": "Cycle started."
}

Fields

  • type: Start / Single / Stop command.
  • latched: Auto-restart when a cycle completes.
  • durations: Extend/retract per piston (seconds).
  • remaining_ms: Time left in the active stage.
  • cycle_count: Completed cycles.
  • message: Optional info text.

Runtime flow

HMI writes a command, client reads and executes; each step is written back to the bus.

  • 1. Command: HMI collects durations, writes write_master_command.
  • 2. Execution: Client steps through extend / retract stages (ms resolution).
  • 3. State: Publishes with write_slave_state (active piston, remaining, latch, message).
  • 4. Latch: If latched=true, waits 0.5 s and restarts.
  • 5. Visualization: Tkinter signal graph or Streamlit panel tracks live signals.

Threading / timing

  • Client loop: 10 ms cadence to read commands and advance the sequence.
  • HMI: Tkinter after(100ms) or Streamlit page refresh.
  • Fake bus: threading.Lock-protected mailbox.

Interfaces & logging

Two HMI options share the same bus and client.

  • Tkinter: Start/Stop/Single buttons, duration spinboxes, latch indicator, signal graph.
  • Streamlit: Web form, live status panel, side-by-side server/client logs with auto refresh.
  • LogBuffer: Thread-safe buffer; pass logger=LogBuffer() to collect client messages.

Piston sequencing & timing

Each piston runs extend then retract; the client builds a flat list of stages and steps through it.

# piston_client.py
sequence = []
for idx, (ext, ret) in enumerate(durations):
    sequence.append(Stage(idx, "extend", int(ext * 1000)))
    sequence.append(Stage(idx, "retract", int(ret * 1000)))

elapsed_ms = (now - stage_started_at) * 1000
if elapsed_ms >= max(stage.duration_ms, 50):
    stage_index += 1
    publish_state(status="running",
                  direction=next_stage.action,
                  active_piston=next_stage.piston_index,
                  remaining_ms=next_stage.duration_ms)

Master vs slave behavior

  • Master (HMI): stateless UI; writes commands on click; polls state to update widgets.
  • Slave (client): owns latch flag, running flag, stage_index, cycle_count.
  • Latch: if true, waits 0.5s after completion and restarts automatically.
  • Validation: durations sanitized (min 0.05s), empty sequence publishes stopped state.

Backend call stack (Python)

From HMI click to piston motion with the Fake EtherCAT bus in between.

HMI button -> _collect_durations()
           -> bus.write_master_command()
Client loop -> pop_slave_command()
           -> _apply_durations() -> _build_sequence()
           -> _start_cycle() -> publish_state("running")
Process cycle -> stage advance -> publish_state(...)
HMI poll -> read_slave_state() -> update UI / logs
